## Prerequisites

Before using the Obsidian REST API skill, ensure you have:

1. **Obsidian** installed (desktop application)
2. **Local REST API plugin** installed and enabled
3. **API key** generated by the plugin
4. **Plugin running** (active on configured port)

## Installing the Plugin

If the Local REST API plugin is not yet installed:

1. Open Obsidian
2. Go to **Settings** (gear icon)
3. Navigate to **Community plugins**
4. Click **Browse** and search for "Local REST API"
5. Install the plugin by **coddingtonbear**
6. Enable the plugin in your Community plugins list

## Finding Your API Key

The Local REST API plugin automatically generates an API key when installed.

### Locate the API Key

1. Open Obsidian **Settings**
2. Navigate to **Community plugins** → **Local REST API**
3. Look for the **API Key** section
4. Copy the displayed key (typically a 64-character hexadecimal string)

Example API key format:
```
cec62292818a7c800b31a5e5cbe2e459141f03716e7fafd1d37130b27c574b29
```

### Generate a New Key

If you need to regenerate the API key:

1. In plugin settings, click **Generate New API Key** or similar button
2. Copy the new key immediately
3. Update your configuration with the new key

**Warning:** Regenerating invalidates the previous key. Any systems using the old key will need updating.

## Configuration

### Option 1: Environment Variable (Recommended)

Store your API key in a `.env` file for secure, reusable access.

**Add to your `.env` file:**
```bash
obsidian_api_keys=cec62292818a7c800b31a5e5cbe2e459141f03716e7fafd1d37130b27c574b29
```

**Usage in requests:**
```bash
curl -H "Authorization: Bearer ${obsidian_api_keys}" \
     http://localhost:27123/vault/
```

The environment variable approach:
- Keeps credentials out of code
- Works across multiple scripts
- Can be easily updated centrally
- Prevents accidental credential exposure

### Option 2: Direct Usage

For one-off requests or testing, you can use the key directly:

```bash
curl -H "Authorization: Bearer YOUR_API_KEY_HERE" \
     http://localhost:27123/vault/
```

**Not recommended for production or repeated use.**

### Base URL Configuration

The default base URL is:
```
http://localhost:27123
```

**Default Port:** 27123

**To verify your port:**
1. Go to plugin settings in Obsidian
2. Check the **Port** or **Server** section
3. Use the displayed port number

**Custom port example:**
If your plugin runs on port `27124`:
```bash
curl -H "Authorization: Bearer ${obsidian_api_keys}" \
     http://localhost:27124/vault/
```

**HTTPS Configuration:**

The plugin can use HTTPS with a self-signed certificate:

1. Enable HTTPS in plugin settings
2. Note the HTTPS port (typically 27124)
3. Use `-k` flag with curl to skip certificate verification:

```bash
curl -k -H "Authorization: Bearer ${obsidian_api_keys}" \
     https://localhost:27124/vault/
```

For production use, configure proper SSL certificates.

## Connection Verification

After configuration, verify the connection works:

### Test 1: Basic Connectivity

Check if the server is running:

```bash
curl http://localhost:27123/
```

**Expected response:** JSON with server status or basic vault information.

**If this fails:**
- Verify Obsidian is running
- Check the plugin is enabled
- Confirm the port number

### Test 2: Authenticated Request

Test with your API key:

```bash
curl -H "Authorization: Bearer ${obsidian_api_keys}" \
     http://localhost:27123/vault/
```

**Expected response:** Vault information or file listing.

**If you get 401 Unauthorized:**
- Verify API key is correct
- Check for extra spaces or newlines in key
- Ensure Bearer token format is correct

### Test 3: Read a Known Note

Test reading a specific note (replace with a note that exists in your vault):

```bash
curl -H "Authorization: Bearer ${obsidian_api_keys}" \
     http://localhost:27123/vault/README.md
```

**Expected response:** The markdown content of the note.

**If you get 404 Not Found:**
- Verify the note exists in your vault
- Check the path and filename
- Ensure file extension `.md` is included

## Security Notes

### API Key Security

- **Full vault access:** The API key grants complete read/write access to your entire vault
- **Treat like a password:** Never commit API keys to version control
- **Use `.env` files:** Keep credentials in `.env` and add to `.gitignore`
- **Rotate regularly:** Consider regenerating keys periodically

### Network Security

- **Localhost only:** By default, the API only accepts connections from localhost (127.0.0.1)
- **Firewall:** Ensure your firewall doesn't expose the API port externally
- **HTTPS recommended:** For any non-localhost access, use HTTPS

### Best Practices

1. **Never hardcode API keys** in scripts or code
2. **Use environment variables** for credential storage
3. **Limit scope:** Only share API keys with trusted applications
4. **Monitor usage:** Check Obsidian logs for unexpected API activity
5. **Keep Obsidian updated:** Install security patches promptly

## Troubleshooting

### Connection Refused

**Symptom:** `curl: (7) Failed to connect to localhost port 27123: Connection refused`

**Solutions:**
1. Verify Obsidian is running
2. Check the Local REST API plugin is enabled
3. Confirm the correct port number
4. Restart Obsidian if needed

### 401 Unauthorized

**Symptom:** HTTP 401 response with authentication error

**Solutions:**
1. Verify API key is correct (copy directly from plugin settings)
2. Check `Authorization` header format: `Bearer <key>`
3. Ensure no extra spaces or newlines in the key
4. Verify the key hasn't been regenerated

### 404 Not Found

**Symptom:** HTTP 404 when accessing a note

**Solutions:**
1. Verify the note exists in your vault
2. Check the path is correct (relative to vault root)
3. Ensure `.md` extension is included
4. Check for typos in the filename
5. Note paths are case-sensitive on some systems

### Empty Response

**Symptom:** Request succeeds but returns empty content

**Solutions:**
1. Verify the note isn't actually empty
2. Check if you're using the correct endpoint
3. Try with `-v` flag to see full response: `curl -v ...`
4. Ensure the Accept header matches expected format

### Plugin Not Responding

**Symptom:** Plugin was working but stopped responding

**Solutions:**
1. Restart Obsidian completely
2. Disable and re-enable the Local REST API plugin
3. Check Obsidian console for error messages (Help → Toggle Developer Tools)
4. Verify no conflicting plugins are installed
5. Check Obsidian logs for errors

### HTTPS Certificate Issues

**Symptom:** SSL certificate verification errors with HTTPS

**Solutions:**
1. Use `-k` flag to skip verification (testing only): `curl -k ...`
2. Install the self-signed certificate in your system trust store
3. Configure proper SSL certificates for production use
